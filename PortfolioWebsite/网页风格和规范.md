# 网页风格与设计规范

本文档记录当前 `PortfolioWebsite` 的真实站点实现，用于约束后续页面扩展、Puck 内容搭建、组件排版和视觉调整。本文档描述的是当前线上风格快照，不负责定义尚未落地的未来目标。

## 1. 文档定位

- 以当前源码、当前内容 JSON 和当前页面可见效果为准。
- 页面组织以 Puck 内容页为核心，而不是单独手写模板优先。
- 新增页面、区块或组件时，优先复用现有组件与 token，不重新发明平行风格。
- 新增组件必须同步更新到 `/playground`，确保样式与排版可回看、可比对。
- 任何组件或页面的排版调整都必须遵循统一网格，不允许脱离栅格做自由摆放。

## 2. 站点当前结构

当前公开页面以以下结构为主：

- 首页 `/`：`HeroSection -> ProjectSection -> ProjectSection -> ProjectSection -> HomeEndcapSection`
- 关于页 `/about`：`PortfolioHeroHeader -> RichParagraph -> HighDensityInfoBlock -> ContactFlashlight -> HomeEndcapSection`
- 作品归档 `/works`：`WorksList -> HomeEndcapSection`
- 灯光归档 `/works/lighting-portfolio`：`PortfolioHeroHeader -> LightingProjectCard`
- 作品详情 `/works/[id]`：由 `HeroHeadline`、`RichParagraph`、`HighDensityInfoBlock`、`ContentCard`、`TextSplitLayout`、`BreakdownTriptych`、`ImageSlider`、`ParameterGrid`、`ImagePanel`、`NextProjectBlock` 等区块按 JSON 组合
- 灯光集合详情 `/works/lighting-portfolio/[id]`：同样由 Puck JSON 驱动渲染

当前网站不是“首页、作品页、联系页完全独立设计语言”，而是在统一视觉系统下，通过不同内容块组合出不同页面密度与节奏。

## 3. 全局视觉基础

### 3.1 背景与色彩

- 页面基础背景为深黑色，核心变量来自 `:root`：
  - `--background: #050505`
  - `--foreground: rgba(255, 255, 255, 0.9)`
  - `--surface-elevated: rgba(255, 255, 255, 0.05)`
  - `--surface-muted: rgba(255, 255, 255, 0.03)`
  - `--border-subtle: rgba(255, 255, 255, 0.1)`
- `html` 与 `body::before` 叠加顶部与边缘的径向光晕、纵向渐变，不使用纯平黑底。
- Tailwind 扩展色以当前 token 为准：
  - `textPrimary: rgba(255, 255, 255, 0.75)`
  - `textMuted: rgba(255, 255, 255, 0.45)`
- 白色只用于最高层级标题、交互高亮和关键边线，正文和说明文本默认降到 `textPrimary` 或 `textMuted`。
- 分隔线默认使用 `border-white/10` 到 `border-white/20`，避免硬边。

### 3.2 表面质感

- 卡片、浮层和灯光归档封面优先使用轻量玻璃质感，而不是重投影卡片。
- `glass-panel` 是当前半透明面板标准：
  - 线性渐变背景
  - 细白边
  - `backdrop-filter: blur(24px)`
- 文字高亮更多依靠反差、混合模式和透明度，不依赖大面积发光。

## 4. 栅格与对齐

### 4.1 核心栅格

- 全站统一使用 12 列 `.grid-container`。
- 当前实现不是旧版“移动端 4 列，平板以上 12 列”，而是所有设备统一 12 列。
- `.grid-container` 规则如下：
  - `grid-template-columns: repeat(12, minmax(0, 1fr))`
  - 默认 `gap: 1rem`
  - `1024px+` 时 `gap: 1.5rem`
  - 两侧安全边距为 `4vw`
  - `max-width: 1680px`
  - 测试模式下的网格辅助线必须按“列宽”和 `gap` 分离绘制，每列显示左右边界，不得把 `gap` 误读成列宽的一部分

### 4.2 常用对齐辅助类

- `grid-content`：标准内容区，`col-start-2 col-span-10`
- `grid-narrow`：窄内容区，`col-start-3 col-span-8`
- `grid-full`：全宽内容区，`col-start-1 col-span-12`
- `grid-left`：左侧内容区，`col-start-2 col-span-5`
- `grid-right`：右侧内容区，`col-start-7 col-span-5`
- `grid-sidebar`：右侧信息栏，`col-start-9 col-span-4`
- `grid-offset-center`、`grid-offset-right`：用于三联图或错位信息块

### 4.3 纵向节奏系统

- 当前站点采用“12 列纵向栅格 + 纵向节奏系统”的双轨规则，不采用严格逐行 baseline grid。
- `8px` 是最小节奏单位，处理按钮内边距、标签间距、局部文字组之间的微调。
- `32px` 是主节奏单位，处理区块级 `pt/pb/mt/mb`、分隔线后的进入留白、页面收束和模块交接。
- 区块外轮廓必须回到 `32px` 节奏线上；组件内部优先使用 `16 / 24 / 32 / 48` 这类 `8px` 倍数做微调，仅在标签锁定、次级说明等光学修正场景下允许使用 `4px`。
- 当前全局节奏 token 与工具类如下：
  - `rhythm-section-compact`：`64px / 64px`
  - `rhythm-section-normal`：`96px / 96px`
  - `rhythm-section-spacious`：移动端 `96px / 96px`，桌面端 `128px / 128px`
  - `rhythm-section-hero`：移动端 `128px / 96px`，桌面端 `160px / 128px`
  - `rhythm-block`：区块间距 `128px`
  - `rhythm-block-compact`：区块间距 `96px`
  - `rhythm-divider-top`：带上边线区块的进入留白 `64px`
  - `rhythm-stack-2 / 3 / 4 / 6 / 8`：内部垂直堆叠工具类，对应 `16 / 24 / 32 / 48 / 64px`
- `HeroSection`、页头 Header、`WorksList`、`HomeEndcapSection`、`StatementBlock` 等 section 级组件应优先通过这些 profile 控制纵向节奏，不再直接散写大块 `pt-*`、`pb-*`。
- 全屏图像段落、转场段和特殊叙事段允许破格，但必须在组件语义上明确自己是例外，不允许匿名脱离主节奏。

### 4.4 对齐规则

- 不允许把整页内容简单垂直堆叠后整体居中。
- 标题、正文、补充信息、CTA 必须通过跨列与偏移建立层次。
- 大图内容和文本内容必须共享同一套栅格边界，不能出现图像自由出血、文本另起一套参考线的情况，除非该组件明确就是全屏图像段落。
- 新增区块优先复用已有栅格辅助类，避免在组件里写新的随意 `col-start` 组合。

## 5. 排版系统

### 5.1 字体预设

当前站点排版不再按“Display / Serif / Mono”三段粗略区分，而是通过 `Typography` 组件统一调用以下 4 套预设：

- `sans-body`
  - 拉丁字族：Futura
  - 中文字族：汉仪旗黑
  - 用途：正文、说明、标签、按钮、元信息
- `luna-editorial`
  - 拉丁字族：Luna
  - 中文字族：思源宋体
  - 用途：大型标题、首页主视觉、作品标题、页尾收束标题
- `gothic-editorial`
  - 拉丁字族：ITC Serif Gothic
  - 中文字族：思源宋体
  - 用途：分类、短标签、带编辑感的辅助标题
- `classical-display`
  - 拉丁字族：DM Serif Display
  - 中文字族：思源宋体
  - 用途：导航抽屉菜单的大号菜单字

### 5.2 字号层级

当前字号 token 以 `Typography` 系统为准：

- `caption`：极小型标签、编号说明、索引辅助信息
- `label`：按钮、分类、菜单说明、短标签
- `body-sm`、`body`、`body-lg`：正文与介绍文案
- `title-sm`：区块标题或密集信息块内标题
- `title`：作品列表标题、次级大标题
- `display`：页头主标题
- `hero`：最大级别标题
- `menu`：导航抽屉菜单字

### 5.3 字重与混排

- 字重由预设内部映射控制，不能简单理解成同一套字体粗细。
- 当前系统会按 script 自动拆分拉丁文和中文，再分别应用对应字族、字重、基线偏移和字距。
- 这套机制是当前站点中西文混排一致性的核心，新增文本组件应继续走 `Typography`，不要回退到裸 `h1`、`p` 加 Tailwind 类名的做法。

### 5.4 换行策略

- `wrapPolicy="heading"`：标题类文本，允许平衡换行
- `wrapPolicy="prose"`：正文类文本，强调阅读舒适度
- `wrapPolicy="label"`：标签类文本，默认大写且尽量不换行
- `wrapPolicy="url"`：邮箱、微信等可复制长文本
- `wrapPolicy="code"`：仅用于需要保留换行和长文本断开的场景

## 6. 图片与媒体呈现

### 6.1 图片预设

当前站点的图片尺寸规范统一走 `PresetImage` 和以下预设：

- `ratio-16-9`
- `ratio-21-9`
- `native`

对应的适配模式为：

- `x`：横向优先
- `y`：纵向优先
- `cover`：填满容器

### 6.2 使用原则

- 所有用户可见图片必须保持直角边界，不允许在站点前台对图片容器或图片画面使用任何 `rounded-*` 圆角样式。
- 首页 Hero、详情 Hero、灯光封面、作品列表预览图都依赖图像和深色遮罩叠加来建立氛围。
- 大图上方通常会再覆盖一层或多层黑色渐变，以保证白色标题可读。
- 默认不允许浏览器原生拖拽媒体元素；正式模式下图片、视频、SVG、Canvas 都关闭默认拖拽与选择。
- 新增图片区块必须明确设定预设比例和 fit mode，不能把图片当作无约束内容直接塞入布局。

## 7. 页面级风格规范

### 7.1 首页

- 首页是“沉浸式入口 + 三段重点项目 + 收束 CTA”，不是传统简历式首页。
- `HeroSection` 采用全屏大图、左大右小的信息分布，并使用 `rhythm-section-hero` 控制首屏落版位置；桌面端文案应与居中的图片比例舞台对齐，而不是跟随整个 section 的最底部。
  - 比例舞台采用“图片比例优先 + 最小高度保护”的策略，避免桌面窗口比例突然变窄时首屏塌成过薄的横条。
  - 首页主标题使用 `luna-editorial`，副标题短标签使用 `gothic-editorial`，右侧说明与 CTA 使用 `sans-body`。
- Hero 背景存在轻微视差和缩放，不做夸张运动。
- `ProjectSection` 使用整屏或接近整屏的项目段落，标题压在图像之上，通过左右交替形成节奏，文字落版应回到 `rhythm-section-normal`。
  - 项目标题下方的锁线宽度应跟随标题可见宽度，而不是使用固定百分比。
- 首页收束区 `HomeEndcapSection` 使用中心式收束，但仍服从栅格，不是任意居中漂浮；该区块默认属于 `rhythm-section-spacious`。

### 7.2 关于页

- 当前联系信息已经并入 `/about`，因此关于页同时承担自述、方法论和联系页职责。
- 页首 `PortfolioHeroHeader` 延续站点统一的大标题页头语言，不另起一套风格；页头 section 默认属于 `rhythm-section-hero`。
- `RichParagraph` 用于较长段落的个人叙述，内容宽度收窄，强调阅读节奏。
- `HighDensityInfoBlock` 用三列密度递进的方式组织背景、方法和结果，而不是单列长文。
- `ContactFlashlight` 是关于页最强交互特征：深浅两层文字叠加，通过鼠标遮罩揭示亮层文本；外层 section 仍需回到主节奏线。
- 联系方式使用 `copyable-contact`，即使在正式模式下也允许复制。

### 7.3 全部作品页

- 当前 `/works` 是档案式索引，不是卡片瀑布流。
- `WorksList` 使用横向分隔线切出每个条目，标题、编号、分类与描述分层明显；列表外层默认属于 `rhythm-section-normal`。
- `WorksListEntry` 的默认状态是描边大标题，激活后显示图像预览与右侧描述。
- 桌面端通过 hover 激活条目；移动端通过视口中心感应区自动展开当前条目。
- 列表页的图像只在激活时显现，不做永久缩略图平铺。

### 7.4 灯光归档页

- `ImageSlider` 必须显示可见标题，明确告诉用户当前在比较什么；底部左右标签必须贴齐图片左右边界，不允许右侧标签停在中线附近。
- `PortfolioHeroHeader` 用于灯光归档首页标题，语气更像 curated index，而不是项目详情头图，页头节奏与其他 Header 保持一致。
- `LightingProjectCard` 是当前灯光归档的核心呈现方式：
  - 外层使用 `glass-panel`
  - 封面图占主导
  - 顶部显示集合编号与进入提示
  - 允许轻微 hover 缩放，不使用复杂翻转或多层卡片动效

### 7.5 作品详情页

- 详情页不是统一模板复刻，而是围绕项目内容使用多个拆解型区块组合。
- 当前常见区块组合包括：
  - `HeroHeadline`
  - `RichParagraph`
  - `HighDensityInfoBlock`
  - `BreakdownHeadline`
  - `ContentCard`
  - `TextSplitLayout`
  - `BreakdownTriptych`
  - `ImageSlider`
  - `ParameterGrid`
  - `ImagePanel`
  - `NextProjectBlock`
- 详情页的视觉目标是“项目拆解文档”，强调阶段、结构、证据和结果，而不是单纯展示一串美术图。

## 8. 交互与动效规范

### 8.1 滚动

- 全站使用 Lenis 做平滑滚动。
- 平滑滚动是默认体验的一部分，但不能影响可访问性或页面焦点管理。

### 8.2 自定义光标

- 在支持高精度指针且用户未开启减少动态效果时，站点隐藏系统光标并启用自定义光标。
- 光标基础形态为白色圆点，使用 `mix-blend-mode: difference`。
- 悬停文本和交互元素时会扩大；磁吸状态会产生更强的聚焦反馈。
- 触摸设备、小屏幕、减少动态偏好，以及后台编辑/Font Lab 场景下必须回退到系统光标。

### 8.3 导航

- 当前站点导航为固定在右上角的 `MENU` 按钮，不是传统顶部横向导航栏。
- 打开后为右侧抽屉式菜单，带模糊遮罩和大字号菜单文字。
- 菜单项使用 `classical-display`，当前活动项有细竖线指示。
- 导航必须保持键盘焦点圈定和 `Escape` 关闭能力。

### 8.4 页面动效

- 首页 Hero、项目段落、作品列表预览、关于页探照灯都使用 Framer Motion，但整体方向仍然克制。
- 当前站点的动效原则是“慢、轻、可感知”，而不是频繁弹跳或过度加速。
- 动效主要承担以下任务：
  - 建立图像与文字的前后景关系
  - 强化滚动中的节奏切换
  - 提供 hover 或焦点反馈
  - 放大当前阅读位置
- `StatementBlock` 这类转场区块除了控制视觉高度，还应声明自己属于 `small / medium / large` 哪一档节奏型区块。

### 8.5 站点模式

- `data-site-mode="normal"` 是当前正式展示模式：
  - 全局禁止文字选择
  - 媒体元素禁止默认拖拽
  - 保持更强的画廊式沉浸感
- `data-site-mode="testing"` 是当前内部测试模式：
  - 允许文本选择
  - 选中效果为高反差白底黑字
  - 默认显示极淡的竖向调试网格细线，固定在页面文档坐标中，用于排查列对齐与安全边距问题
  - 便于校对文案、复制内容和进入内部工具
- `copyable-contact` 属于正式模式下的例外能力，只用于邮箱、微信等需要被复制的联系信息。

## 9. 内容搭建规范

### 9.1 内容来源

- 页面内容以 `content/pages/**/*.json` 为单一内容源。
- 路由通过 `renderPuckPage` 渲染 JSON，不直接在页面文件里硬编码内容主体。
- 首页、关于页、作品归档、灯光归档和详情页都遵循这套内容驱动方式。

### 9.2 新增内容块的要求

- 所有图片类内容块都必须遵循直角图片规则；如果区块含有对比滑块，还必须提供可见标题和左右边界对齐的标签。
- 优先在现有 Puck 组件类别中扩展：
  - 基础布局
  - 作品展示
  - 灯光作品特供
  - 关于与联系
- 新增区块必须说明自己属于哪类页面节奏，不允许出现语义不明的“万能区块”。
- 所有文本必须优先接入 `Typography` 系统。
- 所有图片必须优先接入 `PresetImage` 体系。
- 新区块的默认排版必须直接落在统一 12 列网格上。

### 9.3 页面编排原则

- 首页负责筛选和导向，不承担完整简历功能。
- 关于页负责方法、身份与联系信息整合。
- `/works` 负责索引，不承担完整项目叙事。
- 详情页负责拆解与证明，应使用多段结构而不是单段长文。
- 灯光归档页优先展示系列氛围与集合入口，不提前展开完整分析。

## 10. 当前已知备注

- `/contact` 当前公开行为是重定向到 `/about`，不再是独立联系页。
- `content/pages/contact.json` 仍然存在，但当前公开路由不直接渲染该内容。
- `/works` 页尾 `HomeEndcapSection` 的按钮目前写的是 `/contact`，用户实际会被带到 `/about`。
- 更新本规范时，正文按当前用户可见行为书写；这类内容不一致只记录在备注，不反向包装成正式规范。
