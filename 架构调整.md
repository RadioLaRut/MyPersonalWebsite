# 个人作品集网站 - 架构优化与重构任务

## 项目背景

这是一个基于 Next.js 14 的个人作品集网站项目，目前处于**功能基本完成阶段**，正在逐步填充内容和添加新功能。项目计划部署到 Vercel + CloudFlare，后续可能迁移到腾讯云。

## 当前核心问题

1. **严重的文件重复和混乱**
   - 简历 PDF 文件存在 8+ 个副本，分散在多个位置
   - 存在 `PortfolioWebsite/PortfolioWebsite` 异常嵌套目录（需先检查内容再决定处理方式）
   - public/images 目录已达 436MB，包含大量占位图片

2. **性能问题**
   - 页面加载速度慢，需要优化
   - 大量未优化的图片资源

3. **可维护性问题**
   - 文件组织混乱，查找困难
   - 担心未来添加内容时会更加混乱

## 项目特殊要求（必须遵守）

### 关键约束

1. **保留所有网页效果**：这是一个注重视觉质量的作品集网站，**绝对不可为了优化而删除任何视觉效果或降低画质**
2. **不破坏现有功能**：所有页面路由和交互必须保持正常工作
3. **渐进式重构**：每次改动后都要验证功能完整性，不接受大规模破坏性改动

### 资源管理策略

**根目录 vs PortfolioWebsite 目录：**
- **根目录的 `作品集` 文件夹**：仅供参考使用，不参与网站构建
- **PortfolioWebsite 目录**：实际的 Next.js 项目，所有网站使用的资源应在此目录内
- **PortfolioWebsite/作品集 目录**：需要你先检查内容，确认是否有用后再询问我

**简历 PDF 文件：**
- 目前仅作为参考文档，不需要在网站中使用
- 未来可能提供下载功能，但不是现在需要考虑的（会通过添加组件实现）
- 清理重复副本，只保留必要的版本

**PRD 和设计文档：**
- `stitch_silent_editorial_prd` 等目录是设计稿/PRD 文档
- **必须保留在项目根目录**作为参考，不要移动或删除

### 图片资源管理

1. **当前占位图可以保留**：后续会手动替换为压缩优化后的图片
2. **需要建立图片命名规范**：为未来的资源管理制定清晰的规则
3. **支持现代图片格式**：规划 WebP 等格式的使用，但要注意：
   - 这是注重图片质量的作品集
   - 不得盲目压缩导致画质损失
   - 优化应该是无损或接近无损的
4. **暂不考虑多尺寸响应式**：当前阶段不需要 @1x/@2x/@3x 等多尺寸版本

### 技术细节

**代码引用方式：**
- 需要你先检查代码中如何引用静态资源（相对路径/绝对路径）
- 重构后必须更新所有引用，确保不破坏功能

**Next.js 配置：**
- 当前 next.config.mjs 配置较简单，可能需要优化
- 考虑 Vercel 部署的自动优化特性

**性能优化：**
- 检查是否正确使用 Next.js Image 组件
- 评估代码分割和懒加载的必要性
- 但优先级是：可维护性 = 性能，两者平衡

## 执行任务流程

1. **清理重复文件**
   - 删除确认无用的重复文件
   - 保留单一可信源

2. **重组目录结构**
   - 移动文件到目标位置
   - 更新所有代码引用

3. **优化配置文件**
   - 更新 next.config.mjs
   - 优化 middleware.ts（如需要）

4. **每个步骤后验证**
   - 运行 `npm run dev` 确认开发服务器正常
   - 检查关键页面是否正常显示
   - 验证所有交互效果是否保留

### 第四阶段：验证和文档

1. **全面测试**
   - 测试所有页面路由
   - 验证图片加载
   - 确认视觉效果完整

2. **性能验证**
   - 对比优化前后的加载速度
   - 检查构建产物大小

3. **提供维护指南**
   - 新的目录结构说明
   - 图片资源添加规范
   - 后续维护建议

## 重要提醒

- 不要未经我同意就大规模移动或删除文件
- 不要为了性能优化而牺牲视觉效果
- 不要删除或移动 PRD 文档目录
- 必须在每个步骤后验证功能完整性
- 必须在执行重构前向我展示完整方案
- 必须保持代码的可读性和可维护性
